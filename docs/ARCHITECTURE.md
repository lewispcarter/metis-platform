# Architecture

Metis Platform is a modular monolith with event-driven internals.

## Core Flow

Operational Event → Classification → Workflow → Assignment → Communication → Acknowledgment → Escalation → Resolution → Audit

## Core Modules

- Event Module
- Workflow Module
- Communication Module
- Personnel Module
- Assignment Module
- Audit Module
- Identity Module
- AI Decision Module
- Observability Module

## Security Boundary

Identity is centralized in `IdentityModule`.

The current build supports:

- Clerk bearer-token verification through `ClerkAuthProvider`
- local development auth headers for bootstrap workflows
- normalized `AuthenticatedActor` context
- tenant context through `@CurrentTenant()`
- global RBAC enforcement through `RbacGuard`

## API Contract Visibility

OpenAPI documentation is generated by Nest Swagger and mounted at `/api/docs`.

## Internal Architecture Rules

1. Controllers never trust client-supplied tenant scope when authenticated tenant context exists.
2. Domain modules communicate through service boundaries and domain events.
3. Async operational work runs through Redis/BullMQ queues.
4. Critical state transitions create audit records.
5. External providers stay behind adapter/boundary services.

## Identity Sync Boundary v10

The platform now treats Clerk or development auth as an external identity source, not as the internal source of truth. Each authenticated request is normalized by `AuthBoundaryService`, then synchronized by `PlatformUserService` into platform-owned database records:

- `organization`
- `role`
- `user`
- `organization_membership`

Operational modules use the database-backed `platformUserId` for audit attribution. Provider identifiers remain available as `providerUserId`, but domain logic should not depend on Clerk-specific identifiers.

Identity flow:

```text
Bearer token or development headers
→ ClerkAuthProvider/AuthBoundaryService
→ PlatformUserService sync
→ request.user + request.tenant
→ RBAC Guard
→ tenant-scoped controllers
→ audit attribution
```

This preserves future portability to Auth0, enterprise SSO, or government-controlled identity providers.

## v11 Identity and Activity Layer

The platform now separates raw audit storage from operator-visible activity. `AuditService` remains the immutable compliance boundary. `ActivityModule` maps audit rows into a tenant-scoped operational activity feed.

Core mutation services use `AuditService.recordMutation(actor, organizationId, input)` so user-initiated changes are attributed to the database-backed platform user id while queue/background work remains explicitly system-attributed.

Admin user management is handled through `UserModule`, which can update platform roles, membership roles, and user lifecycle status under RBAC-protected endpoints.

## v0.12 Workflow Execution Hardening

The platform now includes deterministic workflow run and workflow task state machines. Workflow tasks are persisted as first-class operational checkpoints so every long-running workflow can expose step-by-step execution state to operators, supervisors, and auditors.

New workflow execution rules:

- Workflow runs can only move through approved lifecycle transitions.
- Workflow tasks can only move through approved lifecycle transitions.
- Task start/completion/failure timestamps are stored in PostgreSQL.
- Failed task transitions preserve a failure reason for operational review.
- Coverage workflow execution now creates task checkpoints for coverage request creation, candidate discovery, and candidate outreach.

## v0.12 Communication Provider Boundary

Outbound communications now flow through a provider registry. Workflow logic remains provider-agnostic and does not call Twilio, email providers, or local simulators directly.

Provider boundaries:

- `TwilioCommunicationProvider` owns SMS and voice boundaries.
- `EmailCommunicationProvider` owns email boundaries.
- `NoopCommunicationProvider` supports deterministic local development and dashboard/local simulation.
- `CommunicationProviderRegistry` selects the correct provider by channel.

This keeps operational workflows portable across providers and prepares the system for production telephony/email integration without rewriting workflow logic.

## v0.12 Escalation Timer Policy

Escalation policy evaluation is now deterministic and isolated in `escalation-policy.ts`.

Initial policy behavior:

- Acknowledgment timeout escalates immediately.
- Communication failure escalates immediately.
- S1 critical events escalate after 5 minutes unresolved.
- Urgent or critical events escalate after 10 minutes unresolved.
- Unresolved events can trigger scheduled rechecks.

This gives the platform a transparent escalation foundation before database-backed configurable policies are introduced.


## v16 Update — Provider Routing & Settings Administration

This build adds administrator-controlled webhook route management and provider configuration surfaces. Tenant administrators can now register inbound provider addresses, disable routes, view provider readiness metadata, and upsert SMS/voice/email provider configuration records through protected settings endpoints. Public inbound webhooks still resolve tenant scope through active route records and reject unrouted production traffic.
